appledoc

Objective-C documentation set generator.

General comment rules

Multiple line comments

appledoc only handles multiple line comments if they start with slash and double star and end with standard star and slash: 

/** Some comment */
Lines may optionally start with whitespace and a single star char, like this: 
/** Star prefixed comment
 * spreading over multiple
 * lines
 */
In such case, the prefix star is automatically discarded. To make transition from headerdoc a bit simpler, appledoc also accepts headerdoc style multiline comments: 
/*! Comment */

Single line comments

appledoc also handles single line comments that start with three slashes. Two or more single line comments found in consequtive lines are grouped together into a single comment. 

/// Single line comment spreading
/// over multiple lines

Common comment rules

Any comment may optionally include delimiter lines. A delimiter line is any combination of 3 or more chars from the following set: "!@#$%^&*()_=+`~,<.>/?;:'\"-". Such lines are ignored: 

/// ---------------------------------
/// comment
/// ---------------------------------

Formatting text

appledoc has been designed to keep comments in source code as readable as possible. In order to achieve this, it uses Markdown-like syntax and extract as much information from the comment surrounding context. Internally it uses Discount for processing Markdown, which supports a set of extensions to original Markdown; check the link for more details.

Emphasis

Text can be emphasized with the following syntax:

  • Text wrapped within stars (*text*) becomes strong: text.
  • Text wrapped within underscodes (_text_) becomes emphasized: text.
You can include the marker itself in the middle of the text, so this *some * text* is converted to: some * text. You can nest strong and emphasized formatting directives, so _*text*_ or *_text_* becomes emphasized strong text like this: text.

Code spans

As you're documenting source code, chance is you might want to format text as code. Any text wrapped within backtick quotes (`text`) is converted into code span: text. Note that you can't nest emphasis with code!

Links and cross references

Links to web pages and other URLs

Any valid URL address starting with http://, https://, ftp://, file:// or mailto: is automatically converted to a link in generated HTML. Any type of link - URL or cross references described below - can optionally be embedded within less than and greater than symbols like this: <http://gentlebytes.com>.

Links to classes, categories and protocols

Any word that matches known class, category or protocol name is automatically converted to cross reference link to that object. Assuming complete documentation contains class GBClass, it's extension GBClass(), NSError(GBError) extension and protocol GBProtocol, the following text will automatically convert text: 

This text contains links to:
class GBClass, it's extension GBClass(), 
category NSError(GBError) and 
protocol GBProtocol.
Into something like:
This text contains links to: class GBClass, it's extension GBClass(), category NSError(GBError) and protocol GBProtocol.

Links to local members

Any word that matches method or property selector name of "current" class, category or protocol is automatically converted to cross reference link to that member. Assuming current object contains method runWithValue:afterDelay and property value, the following text: 

This text contains links to:
method runWithValue:afterDelay: and
property value.
will automatically convert to something like:
This text contains links to: method runWithValue:afterDelay: and property value.

Links to remote members

In fact, the only time you need to specifically mark text as a link is to create cross reference to methods or properties of another object (i.e. remote object). Assuming complete documentation contains class GBClass which has method runWithValue:afterDelay: and property value, the following text: 

This text contains links to:
method [GBClass runWithValue:afterDelay] and
property [GBClass value]
Will convert to something like:
This text contains links to: method [GBClass runWithValue:afterDelay] and property [GBClass value]

Custom link descriptions

You can also use Markdown like link syntax. This gives you the freedom of using arbitrary description. For example: 

For more info check [this page](http://gentlebytes.com)
or [this one](http://gentlebytes.com "title").
is converted to something like:
For more info check this page or this one.
The address is checked for object cross reference, so you can also do this: 
See [this class](Class) or [this method]([GBClass method:]) for details.
As long Class and [GBClass method:] are valid cross references, this is converted to:
See this class or this method for details.
You can also use reference type links like this: 
See [this class][1].

[1]: Class
Which is properly converted to:
See this class.

Paragraphs

Text is grouped together into paragraphs. A paragraph is simply any number of consecutive lines of text, separated by one or more empty lines: 

First paragraph.

Second paragraph with lot's of
text spread accross multiple lines.

And another paragraph.
Paragraph may contain normal or emphasized text, code spans, links or cross references. Additionally, paragraphs may include blocks: ordered or unordered lists, specially formatted blocks of text and example blocks. Although these items are separated from preceeding paragraph by an empty line, they are considered part of the paragraph!

The first paragraph of each comment is automatically used as short description for the element the comment is describing.

Unordered lists

Unordered lists use -, + or * markers in front of each item: 

- First item.
- Second item with lot's of text
  spread across multiple lines.
- Third item.
Each list item's description is parsed as a paragraph, so all paragraph rules apply. If the list is preceeded by another paragraph, an empty line must be inserted and if followed by another paragraph, it must also be delimited by an empty line: 
Some paragraph.

- First item.
- Second item.

A different paragraph.
This code creates two paragraphs: first one with text "Some paragraph." and given list and second paragraph with text "A different paragraph." If you want to create to separate consequtive lists, delimit them with an empty line like this: 
- First list first item.
- First list second item.

- Second list first item.
- Second list second item.
You can nest lists by prefixing them with tabs like this: 
- Item 1
	- Item 1.1
	- Item 1.2
		- Item 1.2.1
	- Item 1.3
- Item 2
You can combine ordered and unordered lists when nesting.

Ordered lists

Ordered lists use a number followed by a dot as a marker in front of each item: 

1. First item
2. Second item
Numbers don't need to be consequtive, you can even use the same number for all items; appledoc will simply number the items by their order. All rules for unordered lists also apply for ordered lists: separation with empty lines, nesting etc., so it won't be repeated here.

Important information

You mark a paragraph as important with @warning directive. It expects a standard paragraph for the description. For example, this is how you can acchieve effect Apple uses for such blocks: 

@warning *Important:* Sending this message before `readValues:fromFile:`
    will result in unpredicted results, most likely as runtime exception!
As you can see, you can spread the description into multiple lines but any empty line will start a new paragraph.

Bug information

If you'd like to make a paragraph even more emphasized than a @warning, you can use @bug directive. It works just like @warning, so see description there for details.

Examples and code blocks

If you want to mark a block of text as an example, simply prefix each line with a tab or 4 (or more) spaces. appledoc will automatically convert consequtive blocks of tab prefixed lines into a single code block. Similar to lists and other paragraph blocks, example section needs to be delimited from preceeding and following paragraph with an empty line: 

Normal paragraph text

	First line of example
	Second line of example

Another paragraph.
The first tab is automatically removed from the code block.

Methods and properties description

  • @return <description>: Provides the description of method or property result. The description may be composed of a single paragraph as described above. Instead of @return you can also use @returns or @result.
  • @param <name> <description>: Provides the description of method parameter with the given name. You need to provide description for each parameter or appledoc will log a warning. The description may be composed of a single paragraph as described above.
  • @exception <name> <description>: Provides the description of an exception that may be raised by a method. The name of the exception is given with the first parameter. The description may be composed of a single paragraph as described above.

Miscellaneous

  • @see <name> or @sa <name>: See also (cross reference) link. Although you can provide cross reference links anywhere within the paragraph text, as described above, you need to use these directives to provide related context links for documentation set. The name should follow cross reference guidelines described above.
    Note: Methods and properties keep all cross references in generated HTML, regardless of the referenced object. But when @see is used inside class, category or protocol comment, all cross references to other objects or members are ignored. However any cross reference to a template document is converted into companion guide link in the table below the main title. Oh, and remember, you can use nice descriptions using Markdown syntax, for example: [String Programming Guide](document1).
  • @name <title>: The title of the methods and/or properties group. All methods and properties declared after @name directive will be stored into a group with the given title. These groups are then extracted as tasks. Important: @name must be specified in it's own separate comment preceeding the first method or property comment for which the task is specified! So this would work: 
    /** @name Section title */

/** Method description */
    and this wouldn't: 
    /** @name Section title

Method description */

Examples

An example of hypothetical class that handles writting and reading strings to and from files: 

/** This is a short description of the class.

Long description of the class. Lorem ipsum dolor sit amet, consectetur 
adipiscing elit. Quisque vel tortor eget mauris blandit semper a et 
massa. Integer facilisis dignissim ornare. Duis erat erat, vestibulum 
sit amet molestie a, vulputate et ipsum. Vivamus rhoncus tortor vel 
sapien luctus varius. 

@see NSString(GBReadWriteExtensions)
*/
@interface GBStringReader

///-----------------------------------
/// @name Reading & writting handling
///-----------------------------------

/** Reads the given string from the file with the given name.

@param filename The name of the file.
@return Returns string contents of the file or `nil` if reading fails.
@see writeString:toFile:
@see lastString
@see lastFilename
*/
- (NSString *)readStringFromFile:(NSString *)filename;

/** Writes the given string to the file with the given name.

@param string The string to write to file.
@param filename The name of the file.
@return Returns `YES` on success, `NO` otherwise.
@exception NSException Thrown if any parameter is `nil`.
@see readStringFromFile:
@see lastString
@see lastFilename
*/
- (BOOL)writeString:(NSString *)string toFile:(NSString *)filename;

///----------------------------
/// @name Information handling
///----------------------------

/** Returns the last read or written string. */
@property (readonly) NSString *lastString;

/** Returns the last name of the file read from or written to. */
@property (readonly) NSString *lastFilename;

@end
The methods could also have longer descriptions, but I kept the example deliberatly short to make it more readable. You can see more examples within appledoc source code itself on GitHub.
